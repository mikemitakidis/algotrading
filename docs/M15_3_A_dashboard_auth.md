## §1 — What M15.3.A does

| Surface | Before | After |
|---|---|---|
| Password verify | plaintext `==` against `DASHBOARD_PASSWORD` | bcrypt (cost 12) preferred via `DASHBOARD_PASSWORD_HASH`; plaintext fallback retained during transition |
| Rate-limit on `/api/login` | none | sliding 10-min window, 5 failures → 15-min lockout; auditable |
| Session cookies | no flags set | `HttpOnly=True`, `SameSite=Strict`, `Secure=` env-gated (not unconditional — won't break plain-HTTP login) |
| Session lifetime | infinite | hybrid: 30 min idle + 12 h absolute (configurable) |
| Session rotation | none | session rotates on every login (defeats session-fixation) |
| Flask `secret_key` | derived from password (invalidated on rotation, guessable default) | stable `DASHBOARD_SECRET_KEY` env, with safe transitional fallback |
| CSRF protection | none | per-session token; auto-attached by JS; verified on every state-changing endpoint except `/api/login` |
| Audit log | none | new `auth_events` SQLite table (append-only, closed `kind` set) |

## §2 — Environment variables

| Variable | Default | Purpose |
|---|---|---|
| `DASHBOARD_PASSWORD_HASH` | — | bcrypt hash. **Preferred.** Set via `tools/set_dashboard_password.py`. |
| `DASHBOARD_PASSWORD` | (default `changeme` rejected) | Plaintext fallback during transition. Warning emitted on every successful login that uses this path. |
| `DASHBOARD_SECRET_KEY` | — | Flask session-signing key. Stable across password rotations. Auto-generated by `set_dashboard_password.py`. |

| `DASHBOARD_ACCEPT_PLAINTEXT_EXPOSURE` | unset | Silences the "exposed on 0.0.0.0" startup warning. Set to `yes` only after explicitly accepting the transition-window risk. |
| `DASHBOARD_SESSION_IDLE_MIN` | `30` | Idle session timeout in minutes. |
| `DASHBOARD_SESSION_MAX_HOUR` | `12` | Absolute session lifetime in hours. |
| `LOGIN_LOCKOUT_WINDOW_SEC` | `600` | Sliding-window size for counting failed logins. |
| `LOGIN_LOCKOUT_THRESHOLD` | `5` | Failures within the window before lockout. |
| `LOGIN_LOCKOUT_DURATION_SEC` | `900` | How long a lockout lasts. |

## §3 — Operator runbook: switching to HTTPS via Caddy

**Caddy is NOT installed by M15.3.A.** This section documents the recommended path. The single-line edit to bind the dashboard to `127.0.0.1` only is safe to do once Caddy is in place.

### One-time setup (operator action, ~10 minutes)

```bash
# Install Caddy (root)
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update && sudo apt install -y caddy

# Replace /etc/caddy/Caddyfile with:
#   <your-domain-or-vps-ip>:443 {
#       reverse_proxy 127.0.0.1:8080
#   }
sudo systemctl restart caddy

# Edit /opt/algo-trader/.env:
#   add  DASHBOARD_BIND_HOST=127.0.0.1
#   add  DASHBOARD_HTTPS_MODE=true
# Then:
sudo systemctl restart algo-trader-dashboard.service
```

### Verification

```bash
# From the VPS shell — should succeed:
curl -k https://127.0.0.1:8080/api/health        # via Caddy or direct
curl -s http://127.0.0.1:8080/api/health         # direct localhost still works

# From an external machine — should refuse before Caddy is in place:
curl -s http://<vps-ip>:8080/api/health
```

### Trade-off recorded

Until the operator completes the Caddy switch, the dashboard remains on `0.0.0.0:8080` over plain HTTP. The M15.3.A startup warnings make this state visible. `DASHBOARD_ACCEPT_PLAINTEXT_EXPOSURE=yes` silences the warning for the transition window. **The cutover itself is recorded as a permanent-pending item in `docs/NEXT_WORK_REGISTER.md` so it is not lost.**

## §4 — Setting the dashboard password

```bash
# Interactive (preferred). Never prints the password.
cd /opt/algo-trader
sudo /opt/algo-trader/venv/bin/python3 tools/set_dashboard_password.py
```

What this does:
- Backs up `.env` to `.env.bak.<timestamp>` (preserves the backup forever; not auto-deleted).
- Prompts twice for the password (no echo).
- Computes bcrypt cost-12 hash.
- Writes `DASHBOARD_PASSWORD_HASH=<hash>` into `.env`.
- Generates `DASHBOARD_SECRET_KEY=<random>` if not already set.
- Keeps `DASHBOARD_PASSWORD` line (transition) unless `--remove-plaintext` flag.
- Sets `.env` permissions to `0o600`.
- Restart the dashboard service to pick up the change.

Hard cutover (remove the plaintext fallback):

```bash
sudo /opt/algo-trader/venv/bin/python3 tools/set_dashboard_password.py --remove-plaintext
```

## §5 — Rate-limit policy

- **Trade-off (Q-A.1 approved):** lockout state is in-memory per Flask process. A dashboard restart clears active lockouts. Acceptable because:
  - Every attempt — successful, failed, or locked — writes to `auth_events`, so forensic visibility persists across restarts.
  - bcrypt-verify takes ~250ms (cost factor 12), making brute-force CPU-bound at scale regardless of in-memory state.
  - Single-operator system; the operator notices a lockout fast.
- **Operator override:** to clear an unexpected lockout for your own IP, restart the dashboard service (`sudo systemctl restart algo-trader-dashboard.service`). All sessions remain valid (cookies are signed with the stable `DASHBOARD_SECRET_KEY`); only the in-memory lockout counters reset.
- **Future upgrade:** SQLite-backed lockout state is recorded as a deferred enhancement in `docs/NEXT_WORK_REGISTER.md` (M15.3.A.persist). Implement only if a real brute-force incident materializes.

## §6 — CSRF model

- Token: 32-byte URL-safe base64, generated per session at login time.
- Storage: in `session["csrf_token"]` (Flask-signed cookie).
- Transport: `X-CSRF-Token` HTTP header (preferred) or `csrf_token` form field.
- Verification: constant-time compare against the session-stored token.

**Exempt endpoints** (all other state-changing endpoints require CSRF):

| Endpoint | Reason for exemption |
|---|---|
| `POST /api/login` | No session yet to embed the token in. Protected by rate-limit + bcrypt. |

**Verifying CSRF works from a fresh shell:**

```bash
COOKIE=$(mktemp)
PASSWORD='<your password>'

# 1. Login — receive CSRF token + session cookie.
RESP=$(curl -s -c $COOKIE -X POST http://127.0.0.1:8080/api/login \
            -H 'Content-Type: application/json' \
            -d "{\"password\":\"$PASSWORD\"}")
CSRF=$(echo "$RESP" | python3 -c 'import json,sys;print(json.load(sys.stdin)["csrf_token"])')

# 2. A state-changing call WITHOUT the header fails with 403.
curl -s -b $COOKIE -X POST http://127.0.0.1:8080/api/logout
# Expect: {"error":"csrf_invalid"} HTTP 403

# 3. The same call WITH the header succeeds.
curl -s -b $COOKIE -X POST http://127.0.0.1:8080/api/logout \
     -H "X-CSRF-Token: $CSRF"
# Expect: {"ok":true}

rm -f $COOKIE
unset PASSWORD CSRF RESP
```

## §7 — `auth_events` schema reference

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK AUTOINCREMENT | Monotone — comparison-safe for ordering. |
| `ts_utc` | TEXT NOT NULL | ISO-8601 with timezone (microsecond precision). |
| `kind` | TEXT NOT NULL | Closed set — see below. |
| `client_ip` | TEXT | Raw IPv4/v6 (Q-A.2 approved). Empty string if not derivable. |
| `user_agent` | TEXT | Truncated to 200 chars (Q-A.8 approved). |
| `session_id` | TEXT | SHA-256 hash of the raw session cookie (never raw — Q-A.8). |
| `success` | INTEGER NOT NULL | `0` or `1` (CHECK-enforced). |
| `extras_json` | TEXT | App-controlled JSON; never contains passwords or secrets. |

**Closed `kind` set:**
`login_success`, `login_failure`, `login_locked`, `login_unconfigured`, `logout`, `session_rotate`, `csrf_invalid`, `session_expired`.

**Forensic queries:**

```sql
-- Most recent login attempts
SELECT ts_utc, kind, client_ip, success
FROM auth_events
ORDER BY id DESC
LIMIT 50;

-- Failed-login pattern by IP in the last hour
SELECT client_ip, COUNT(*) AS attempts
FROM auth_events
WHERE kind = 'login_failure'
  AND ts_utc > datetime('now', '-1 hour')
GROUP BY client_ip
ORDER BY attempts DESC;
```

## §9 — Rollback

Single-step. The protected runtime files are byte-identical so a rollback only reverts the dashboard surface.

```bash
cd /opt/algo-trader
sudo git fetch origin main
sudo git reset --hard <pre-M15.3.A commit hash>   # currently 60281c4
sudo systemctl restart algo-trader-dashboard.service
```

The `auth_events` SQLite table created by M15.3.A is additive — leave it in place after rollback (orphan audit rows are useful evidence; the rolled-back code simply ignores them). `.env` modifications (`DASHBOARD_PASSWORD_HASH`, `DASHBOARD_SECRET_KEY`) are also safe to leave; the rolled-back code uses only `DASHBOARD_PASSWORD`.
